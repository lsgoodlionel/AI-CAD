# Phase E3-0 建模覆盖缺口审计报告

> 2026-07-16 ｜ 验证项目:上海大歌剧院(project 9188e163…,2309 图)
> 目的:E3「先测后改」——用真实数据判定外墙/围护/地下室/幕墙/钢构缺口的可恢复性,
> 避免按错误假设投入多天开发。

## 结论(先说)

**歌剧院几何识别的"易摘果实"已摘完——识别器无 bug,是几何表达与识别假设不匹配。**
剩余缺口(外墙/幕墙/钢构/围护桩)本质是「从线段汤检测圆/符号」的研究级问题,
且高度依赖本套图画法,与需求#8 通用原则相悖。**建议作为独立专项评估,不在 E3 内盲目投入。**

## 关键事实(实测证据)

### 1. 全 PDF,无 CAD 图层

```
歌剧院 2309 图: 100% PDF, 0 DXF
```
→ E3 蓝图原方案「扩 layer_conventions/layer_class_map 图层词表」对本项目**完全无效**
(PDF 无图层)。图层词表仍对 DXF 客户有价值,但不能用歌剧院验证。

### 2. 矢量文字取不到

抽样 3 张(围护体平面/钢立柱平面/钢支撑平面),`geometry_extractor.texts`
(fitz `get_text("words")`)对「钢/围护/幕墙/立柱/支撑/地下」关键词命中**全为 0**
——尽管图名就是「钢立柱平面布置图」。CAD PDF 正文标注是矢量字形,`get_text` 取不到。
→ 靠矢量文字关键词判构件类型**行不通**;文字信号只能来自档案 OCR。

### 3. 识别器无欠费(反直觉但确凿)

「钢立柱平面布置图」实测:
```
scale=0.03528 m/pt | CAPS.columns=2000(远未触顶)
polys=717, lines=19283(其中 17131 条 <0.8m 短线段)
0.2-1.5m 近方(aspect<2)poly 总数 = 13  ←→  实际识别柱数 = 13(完全一致)
0.6-1.5m poly: 近方 2 / 细长 598,且 600 个全是 9+ 顶点(曲线路径,非柱)
```
→ 柱识别器**已抓完所有近方闭合多段线(13/13)**。识别少不是阈值/CAP 问题,
而是**钢柱/围护桩根本不以"近方闭合多段线"表达**,而是圆/短线段簇。

### 4. 缺口的真实形态

- 围护桩:圆(由短弧/线段近似)——非闭合近方 poly,识别器 key 不到。
- 钢立柱:小方框/H 型钢符号,多由独立线段构成——非 poly。
- 外墙/围护体:长线,平行线对识别命中少(7 墙),因线段被打散。
- 恢复它们需:圆检测(Hough/线簇拟合)、短线段聚类成符号、外轮廓环闭合。

## 可选路径与成本

| 路径 | 对歌剧院 | 通用性 | 成本/风险 |
|------|---------|--------|----------|
| A. 图层词表扩幕墙/钢构(原 E3-1) | ❌ 无效(无图层) | ✅ 利 DXF 客户 | 低,但需合成 DXF 验收 |
| B. 攻几何识别(圆/符号/线簇) | ⚠️ 或可提升 | ❌ 偏本套图画法 | **高、多天、脆弱、回报不确定** |
| C. 档案 OCR 文字反哺分类 | ⚠️ 中等 | ✅ 通用 | 中,依赖 OCR 位置关联(同轴网变换层问题) |

## 建议

1. **不在 E3 内盲目做几何攻坚**(路径 B)——研究级、脆弱、违通用原则。
2. 若要提升歌剧院,**先做路径 C**(档案 OCR 文字按位置关联几何 → 补类型标签),
   它同时惠及所有 PDF 项目,且与「档案层单一真相源」架构一致;圆检测(B)视 C 效果再定。
3. **能力边界如实写入手册**:PDF-only 且矢量文字不可提取的项目,构件识别限于
   几何近方/闭合形状;圆形桩、线段符号、幕墙曲面不支持——非缺陷,是方法边界。

## 路径 B 实测结论(2026-07-16,已实现)

栅格化 + OpenCV HoughCircles 圆检测(`core/model3d/circle_detector.py`)对纯 PDF
桩/圆柱有效,双闸防误检(仅平面图 + 仅结构/通用专业)、尺寸下限 0.5m、去重:

| 图 | 圆检测 | 柱数(前→后) |
|----|--------|-------------|
| 围护体平面 | plan 开 | 70 → 356 |
| 钢立柱平面 | plan 开 | 13 → 361 |
| 围护体剖面 | section 关 | 71 → 71(零误检) |

**整机重建确认**:歌剧院全模型 columns **3089 → 5794**(+2705 桩/圆柱),
建模 8 分钟(标高读档案不跑 OCR),无崩溃。结论:**路径 B 对"桩以圆表达"的
PDF 项目有效且可控**;仍待 3D 可视抽查确认落位无异常。

## 路径 C(A1/A2)实现与现状(2026-07-16)

- **A1 每图坐标变换持久化**(migration 031 `drawing_transform`):抽取时复用识别器
  `_detect_scale/_origin_pt` 算出 scale/origin/page_h 落库,让档案 pt 坐标可转米。
  歌剧院实测 705/2309 图有变换(其余无轴线,原点不可靠不落)。
- **A2 档案轴号→3D**(`archive_axes_to_scene` + build_floor_elements 并入):
  档案好标签(1/2/3/A/B)经 A1 变换转米,升级识别路径的 "X" 噪声标签、补覆盖。
  代码完成、49 测试绿、逻辑验证(54 图同时有变换+档案轴号)。
  **现状:整机未显效**——档案轴号 item 依赖 OCR 抽取,而 OCR 289s/图 太慢
  (2309 图约一天),已抽的 ~287 图与"贡献识别轴网的结构图"不重叠。
  **A2 效果随 OCR 回填铺开逐步显现,非代码问题,是数据管线时间问题。**

## E3-4 板/管线(2026-07-16)

- **管线降噪**:短线噪声已被 `_PIPE_MIN_LEN_M` 过滤(实测 0 条 <1m);剩余
  5732 管线为长线,噪声=「任何长线被当管线」(尺寸线/边框/剖切线),
  深度降噪需模式分析(区分管线中线 vs 标注线)、有误删真管线风险 → **不做冒险过滤**
  (同研究级结论)。
- **板补充**:板对楼层堆叠模型应为数十块量级(非几千);现 16 偏低。
  已实现 `ensure_slab_from_columns`——pile 增强后若该图无板,用柱/桩包络补底板
  (基坑楼层得板参与体量/算量;≥8 柱、已有板不覆盖)。深度板恢复(从线段闭合
  楼面外轮廓)仍是线条汤难题,未做。
